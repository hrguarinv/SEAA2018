A Systematic Mapping on Agile Security Requirements Engineering

Abstract

The rapidly changing business environment in which most companies operate is challenging traditional Requirements Engineering (RE) approaches. The term agile RE has been used to define the ‘‘agile way’’ of conducting RE activities. Security is an essential non-functional requirement that tends to be difficult to address in agile development contexts. The overall understanding of how to handle security requirements in agile contexts is still vague. [Objective] The goal of this paper is to characterize the approaches that have been proposed to handle security requirements in agile software projects. [Method] We conducted a systematic mapping study. In total, we identified 21 studies that met our inclusion criteria, dated from 2005 to 2017. [Results] We found the approaches propose solutions that may be divided into four categories: modify agile method, extend the concept of user story to abuser story, introduce a lightweight methodology and develop a software framework. We also identified some limitations of using these approaches related to environment, people, effort and resources. [Conclusion] Our analysis suggests that more effort needs to be put into empirically evaluating the solutions and develop specialized software tools in order to mitigate the identified limitations.

Systematic Mapping Protocol

In this research a systematic mapping study was performed following the guidelines proposed by Kitchenham and Charters [16]. It was possible to identify the need for the review, to specify the research questions, the research string and the inclusion/exclusion criteria.

Metric Description
Coupling Between Objects (CBO) [1] Number of components coupled to one specific component
Fan-in (FANIN) [2] Number of dependencies from other components to a specific component
Fan-out (FANOUT) [2] Number of dependencies from one components to others
Depth of Inheritance Tree (DIT) [1] Maximum depth of a component in the inheritance tree
Number Of Children (NOC) [1] Number of children components of a component
